---
title: "Configure Alita Robot for production webhook delivery"
description: "Switch Alita Robot to production webhook mode using Cloudflare Tunnel for real-time updates, lower resource usage, and firewall-friendly HTTPS delivery."
---

By default Alita Robot uses long-polling: it repeatedly asks Telegram for new updates. Polling is convenient for development because it works anywhere with outbound internet access, but it introduces a 1–3 second delay and consumes more CPU and bandwidth than necessary. Webhook mode inverts the flow — Telegram pushes each update to your HTTPS endpoint the instant it arrives, making it the right choice for production deployments.

## Webhook vs polling

| Feature | Webhook mode | Polling mode |
|---|---|---|
| **Update latency** | Real-time | 1–3 second delay |
| **Resource usage** | Lower CPU and bandwidth | Higher CPU and bandwidth |
| **Setup complexity** | Requires an HTTPS endpoint | Works anywhere, zero setup |
| **Production suitability** | Recommended | Development only |
| **Firewall / NAT** | Works behind NAT via Cloudflare Tunnel | Requires outbound access only |

## Setting up Cloudflare Tunnel

Cloudflare Tunnel gives you a stable public HTTPS URL without opening ports or managing TLS certificates. It is the recommended way to expose Alita Robot's webhook endpoint.

<Steps>
  <Step title="Create a tunnel in Cloudflare Zero Trust">
    1. Go to the [Cloudflare Zero Trust Dashboard](https://one.dash.cloudflare.com).
    2. Navigate to **Networks > Tunnels**.
    3. Click **Create a tunnel** and choose **Cloudflared**.
    4. Name your tunnel (e.g. `alita-telegram-bot`).
    5. Copy the **tunnel token** — the long string shown after `--token` in the install command.
  </Step>

  <Step title="Configure a public hostname">
    In your tunnel's **Public Hostnames** tab, click **Add a public hostname** and set:

    | Field | Value |
    |---|---|
    | Subdomain | `alita-bot` (or any name you prefer) |
    | Domain | Your domain added to Cloudflare |
    | Service | `http://alita:8080` |
    | Path | `/webhook/your-secret` (use your actual `WEBHOOK_SECRET`) |

    The path must match the secret you set in your environment — Telegram will only deliver updates to this exact URL.
  </Step>

  <Step title="Configure environment variables">
    Update your `.env` with webhook settings:

    ```env
    # Bot configuration
    BOT_TOKEN=your_bot_token_here
    OWNER_ID=your_telegram_user_id
    MESSAGE_DUMP=-100xxxxxxxxx

    # Database
    DATABASE_URL=postgres://postgres:password@postgres:5432/alita_robot?sslmode=disable
    REDIS_ADDRESS=redis:6379
    REDIS_PASSWORD=your_redis_password

    # Webhook
    USE_WEBHOOKS=true
    WEBHOOK_DOMAIN=https://alita-bot.yourdomain.com
    WEBHOOK_SECRET=your-random-secret-string-here
    HTTP_PORT=8080

    # Cloudflare Tunnel
    CLOUDFLARE_TUNNEL_TOKEN=eyJhIjoiNzU1...your-tunnel-token-here
    ```
  </Step>

  <Step title="Uncomment the cloudflared service in docker-compose.yml">
    Add or uncomment the `cloudflared` service block in `docker-compose.yml`:

    ```yaml
    cloudflared:
      image: cloudflare/cloudflared:latest
      container_name: alita-cloudflared
      environment:
        - TUNNEL_TOKEN=${CLOUDFLARE_TUNNEL_TOKEN}
      command: tunnel --no-autoupdate run
      restart: unless-stopped
      depends_on:
        - alita
      deploy:
        resources:
          limits:
            memory: 128M
            cpus: "0.1"
    ```
  </Step>

  <Step title="Start the stack">
    ```bash
    docker-compose up -d
    ```

    This starts the bot, PostgreSQL, Redis, and the Cloudflare tunnel agent together.
  </Step>

  <Step title="Register the webhook with Telegram">
    With the bot running, tell Telegram where to send updates:

    ```bash
    curl -X POST "https://api.telegram.org/botYOUR_BOT_TOKEN/setWebhook" \
      -H "Content-Type: application/json" \
      -d '{
        "url": "https://alita-bot.yourdomain.com/webhook/your-secret",
        "secret_token": "your-secret"
      }'
    ```

    Replace `YOUR_BOT_TOKEN`, the URL, and `your-secret` with your actual values.

    A successful response looks like:

    ```json
    {"ok": true, "result": true, "description": "Webhook was set"}
    ```
  </Step>
</Steps>

## Webhook environment variables

| Variable | Description | Default |
|---|---|---|
| `USE_WEBHOOKS` | Set to `true` to enable webhook mode | `false` |
| `WEBHOOK_DOMAIN` | Full public HTTPS URL of the bot (no trailing slash) | — |
| `WEBHOOK_SECRET` | Random secret string; included in the webhook URL path | — |
| `HTTP_PORT` | Port for the unified HTTP server (health, metrics, webhook) | `8080` |
| `CLOUDFLARE_TUNNEL_TOKEN` | Tunnel token from the Cloudflare Zero Trust dashboard | — |

<Warning>
  `WEBHOOK_SECRET` is required when `USE_WEBHOOKS=true`. Without it the webhook endpoint will not be registered correctly and Telegram will not deliver updates. Use a long, random string — at least 32 characters.
</Warning>

## Verifying webhook setup

After registration, check that Telegram can reach your endpoint:

```bash
curl "https://api.telegram.org/botYOUR_BOT_TOKEN/getWebhookInfo"
```

A healthy response includes your URL and shows `pending_update_count` near zero:

```json
{
  "ok": true,
  "result": {
    "url": "https://alita-bot.yourdomain.com/webhook/your-secret",
    "has_custom_certificate": false,
    "pending_update_count": 0,
    "last_error_date": 0,
    "max_connections": 40
  }
}
```

If `last_error_message` appears in the response, Telegram is having trouble reaching your endpoint. Check the `cloudflared` container logs:

```bash
docker-compose logs cloudflared
```

## Switching back to polling

To remove the webhook and return to polling mode:

```bash
# Delete the registered webhook
curl -X POST "https://api.telegram.org/botYOUR_BOT_TOKEN/setWebhook" \
  -d "url="
```

Then update your `.env`:

```env
USE_WEBHOOKS=false
```

And restart the bot:

```bash
docker-compose restart alita
```

<Tip>
  You can also set `DROP_PENDING_UPDATES=true` when switching modes to discard any updates that accumulated while the bot was offline, preventing a flood of stale messages on startup.
</Tip>
